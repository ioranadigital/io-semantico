# CRM Pro — Revisar y activar datos en local

Basado en estas instrucciones, implementa los cambios en el disco local.

---

## OBJETIVO

Revisar que cada módulo del CRM muestra datos reales funcionando en local.
Para los módulos que estén vacíos, crear datos de prueba (seed) en Supabase
y corregir cualquier error de conexión, query o RLS que impida que los datos
aparezcan en pantalla.

---

## 6. PASO 5 — CORREGIR ERRORES DE QUERY (400)

Si una query falla con error 400, las causas más comunes son:

**Vista no existe:**
```sql
-- Verificar que las vistas están creadas
select viewname from pg_views where schemaname = 'public';
-- Debe mostrar: v_project_profitability, v_retainer_status, v_client_timeline
```

Si alguna vista no existe, crearla:
```sql
-- v_retainer_status (más importante para Maintenance)
create or replace view v_retainer_status as
select
  r.id, r.account_id, a.name as client, a.color_hex, a.avatar,
  r.hours_month, r.hours_used as hours_used_field,
  coalesce(sum(tl.hours), 0) as hours_used,
  round(coalesce(sum(tl.hours), 0) / nullif(r.hours_month, 0) * 100, 1) as pct_used,
  r.price_month, r.renew_date, r.alert_pct, r.active
from retainers r
join accounts a on a.id = r.account_id
left join projects p on p.account_id = r.account_id
left join tasks t on t.project_id = p.id
left join timelogs tl on tl.task_id = t.id
  and date_trunc('month', tl.logged_at::date) = date_trunc('month', current_date)
group by r.id, a.name, a.color_hex, a.avatar;

-- v_project_profitability
create or replace view v_project_profitability as
select
  p.id, p.name, a.name as client, p.budget, p.spent,
  coalesce(sum(tl.hours), 0) as hours_logged,
  coalesce((select sum(t2.hours_budget) from tasks t2 where t2.project_id = p.id), 0) as hours_budget,
  round(coalesce(p.spent,0) / nullif(p.budget,0) * 100, 1) as budget_pct,
  p.status, p.due_date
from projects p
join accounts a on a.id = p.account_id
left join tasks t on t.project_id = p.id
left join timelogs tl on tl.task_id = t.id
group by p.id, a.name;

-- v_client_timeline
create or replace view v_client_timeline as
  select account_id, 'ticket' as type,
    title as text, resolved_at as event_date, ref as ref
  from tickets where status = 'Resuelto' and resolved_at is not null
union all
  select account_id, 'hito' as type,
    name || ' — hito completado' as text,
    updated_at as event_date, id::text as ref
  from projects where milestone_done > 0
union all
  select account_id, 'factura' as type,
    ref || ' pagada · ' || amount::text || '€' as text,
    paid_date::timestamptz as event_date, ref as ref
  from invoices where status = 'Pagada' and paid_date is not null
order by event_date desc;
```

**Join incorrecto en la query:**
Si el frontend hace `.select('*, accounts(name)')` y falla, verificar que la FK existe:
```sql
select conname, conrelid::regclass, confrelid::regclass
from pg_constraint
where contype = 'f'
order by conrelid::regclass::text;
```

## 7. PASO 6 — CORREGIR PROBLEMAS DE RENDERIZADO

Si los datos llegan de Supabase pero no se ven en pantalla:

**El módulo muestra SkeletonLoader permanentemente:**
- El `isLoading` nunca cambia a false
- Solución: añadir `finally { setIsLoading(false) }` en el bloque de fetch

**Los datos llegan pero el componente no los renderiza:**
- Revisar que el tipo del dato coincide con la interfaz TypeScript
- Revisar que no hay `undefined` en el acceso a propiedades anidadas (usar optional chaining `?.`)

**Las relaciones (joins) no traen datos:**
Si `.select('*, accounts(name)')` no trae el campo `accounts.name`:
- En Supabase, las relaciones en el select usan el nombre de la tabla referenciada
- Verificar que la FK está correctamente definida en el schema
- Usar la sintaxis exacta: `accounts(name, color_hex)` para múltiples campos

**El donut o gráficos no renderiza:**
- Verificar que recharts y el componente DonutChart están importados correctamente
- Los SVG inline necesitan valores numéricos, no strings. Usar `Number(pct)` o `parseFloat`
